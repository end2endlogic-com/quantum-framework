[[bootstrap-packs]]
=== Bootstrap Packs

BootstrapPacks are the framework mechanism for idempotent runtime setup work that is bigger than declarative seeding.

They are intentionally distinct from migrations and seed packs:

- migrations change database structure and initialization state
- seed packs materialize deterministic datasets
- BootstrapPacks orchestrate operational setup steps across one or more systems

The framework now provides the BootstrapPack runtime, admin APIs, startup runner, and SPI hooks. Product modules contribute actual packs and step handlers.

==== What a BootstrapPack is

A BootstrapPack is a named, versioned sequence of steps:

- `packRef`
- `packVersion`
- `productRef`
- `profileRef`
- ordered `steps`

Each step has:

- `stepRef`
- `description`
- `kind`
- `applyPolicy`
- `dependsOn`
- `config`

At runtime, `BootstrapPackService` resolves a pack from the registry, builds a scope, executes each step through the matching `BootstrapPackStepHandler`, and records a `BootstrapPackRun`.

==== Why BootstrapPacks exist

Seed packs are excellent for baseline data, but some startup and tenant-activation work is not just "load these JSON records":

- calling a product service to ensure a resource exists
- triggering an HTTP integration to an external platform
- validating that a prerequisite already exists
- publishing workflow or solution metadata
- sequencing several actions with dependencies and run history

That is the gap BootstrapPacks fill.

==== BootstrapPacks vs seed packs

[cols="1,2,2", options="header"]
|===
| Question | Seed packs | BootstrapPacks

| Primary job
| Load deterministic baseline data
| Coordinate idempotent operational setup steps

| Input format
| Manifest plus JSON/NDJSON datasets
| Java `BootstrapPackDefinition` plus step configs

| Best for
| Code lists, policies, defaults, reference rows, baseline records
| Calling services, verifying prerequisites, external side effects, multi-step orchestration

| Idempotency model
| Dataset checksums plus natural keys
| Step handler logic plus apply mode and apply policy

| Cross-system work
| Poor fit
| First-class use case

| Dependencies
| Includes/archetypes inside manifests
| `dependsOn` between steps inside one pack

| Typical owner
| Framework or product module shipping data
| Product/application module shipping setup logic
|===

==== When to use one vs the other

Use a seed pack when the desired result is "these records should exist in the tenant realm."

Examples:

- install code lists
- create default policies
- load default feature metadata
- establish baseline survey definitions

Use a BootstrapPack when the desired result is "run these steps to make the environment ready."

Examples:

- ensure an app service has provisioned a workspace
- call an external API to register a tenant
- run assertions before enabling a solution
- publish a workflow package after required seed data exists

Use both together when needed:

- a BootstrapPack step can depend on a `SEED_PACK` step to materialize baseline data first
- later steps can then call services or external APIs that rely on that seeded data

==== Rule of thumb

If you can review the change entirely as data files, prefer a seed pack.

If the change needs code, orchestration, side effects, or conditional runtime checks, prefer a BootstrapPack.

==== Step kinds

The framework model currently defines these step kinds:

- `SEED_PACK`
- `APP_SERVICE`
- `HTTP_CALL`
- `ASSERTION`
- `WORKFLOW_PLATFORM_PUBLICATION`

The enum is shared infrastructure. Actual behavior depends on the `BootstrapPackStepHandler` implementations contributed by the application or product modules on the classpath.

==== Apply modes and policies

BootstrapPack apply mode controls how the pack is being run:

- `APPLY_MISSING`
- `REAPPLY`
- `VALIDATE_ONLY`

Step apply policy describes the intended handler behavior:

- `ONCE_PER_SCOPE`
- `WHEN_MISSING`
- `ALWAYS_IDEMPOTENT`

The framework passes both values into handlers through `BootstrapStepRequest`. Handler authors decide what those semantics mean for a specific step kind.

==== SPI model

Applications contribute packs with `BootstrapPackContributor`:

[source,java]
----
@ApplicationScoped
public class DemoBootstrapContributor implements BootstrapPackContributor {
    @Override
    public Collection<BootstrapPackDefinition> bootstrapPacks() {
        return List.of(new BootstrapPackDefinition(
            "demo-pack",
            "2026.04.0",
            "b2bi",
            "local-demo",
            List.of(
                new BootstrapPackStepDefinition(
                    "ensure-a",
                    "Ensure resource A exists",
                    BootstrapStepKind.APP_SERVICE,
                    BootstrapStepApplyPolicy.WHEN_MISSING,
                    List.of(),
                    Map.of("resource", "A")
                ),
                new BootstrapPackStepDefinition(
                    "ensure-b",
                    "Ensure resource B exists",
                    BootstrapStepKind.APP_SERVICE,
                    BootstrapStepApplyPolicy.WHEN_MISSING,
                    List.of("ensure-a"),
                    Map.of("resource", "B")
                )
            )
        ));
    }
}
----

Handlers implement the work for one or more step kinds:

[source,java]
----
@ApplicationScoped
public class DemoAppServiceHandler implements BootstrapPackStepHandler {
    @Override
    public boolean supports(BootstrapStepKind kind) {
        return kind == BootstrapStepKind.APP_SERVICE;
    }

    @Override
    public BootstrapStepResult execute(BootstrapStepRequest request) {
        return new BootstrapStepResult(
            BootstrapStepStatus.COMPLETED,
            BootstrapStepOutcome.CREATED,
            Map.of("handledStep", request.stepRef())
        );
    }
}
----

==== Admin REST API

The framework exposes admin endpoints at `/admin/bootstrap-packs`:

[cols="1,3", options="header"]
|===
| Endpoint | Purpose

| `GET /admin/bootstrap-packs`
| List available pack definitions

| `GET /admin/bootstrap-packs/pending/{realm}`
| List packs with no successful completed run for the realm

| `GET /admin/bootstrap-packs/history/{realm}`
| Show recorded runs for the realm

| `POST /admin/bootstrap-packs/validate/{realm}`
| Validate all selected packs without applying changes

| `POST /admin/bootstrap-packs/{realm}/{packRef}/validate`
| Validate one pack

| `POST /admin/bootstrap-packs/apply/{realm}`
| Apply all selected packs

| `POST /admin/bootstrap-packs/{realm}/{packRef}/apply`
| Apply one pack
|===

Useful query parameters:

- `filter=pack-a,pack-b`
- `mode=APPLY_MISSING`
- `mode=REAPPLY`
- `mode=VALIDATE_ONLY`

These endpoints are protected for `admin` and `system` roles.

==== Startup application

BootstrapPacks can also be auto-applied at startup by `BootstrapPackStartupRunner`.

Relevant properties:

[source,properties]
----
quantum.bootstrap-pack.enabled=true
quantum.bootstrap-pack.apply.on-startup=false
quantum.bootstrap-pack.apply.realms=system-com,mycompany-com,test-com
quantum.bootstrap-pack.apply.filter=pack-a,pack-b
----

If startup application is enabled, the runner:

1. resolves the target realms
2. resolves the selected packs
3. runs each pack as `APPLY_MISSING`
4. logs failures and pending packs

==== Run history

The framework records `BootstrapPackRun` objects, including:

- pack ref and version
- execution scope such as realm and environment
- per-step status and outcome
- start and finish timestamps

By default, the framework uses an in-memory run repository. That is useful for local development and tests, but products that need durable operational history should plug in a persistent `BootstrapPackRunRepository`.

==== Examples

===== Example 1: Use a seed pack only

Requirement:
Add baseline code lists, policies, and default records for every new tenant.

Best choice:
Use a seed pack.

Reason:
The desired result is purely deterministic data in MongoDB.

===== Example 2: Use a BootstrapPack only

Requirement:
After a tenant is created, call a product service to create a workspace and then verify it exists.

Best choice:
Use a BootstrapPack with `APP_SERVICE` and `ASSERTION` steps.

Reason:
The work is procedural and depends on service logic, not just document inserts.

===== Example 3: Use both together

Requirement:
Seed workflow definitions first, then publish them to an external workflow platform.

Best choice:

- Use a seed pack for the workflow and policy records
- Use a BootstrapPack with a `SEED_PACK` step followed by `WORKFLOW_PLATFORM_PUBLICATION`

Reason:
The first half is declarative data, while the second half is an operational publication step.

==== Choosing the right abstraction

Prefer seed packs by default for tenant baseline data.

Move to BootstrapPacks when:

- the setup spans more than one system
- steps must run in order with dependencies
- you need dry-run validation semantics
- the setup is product-owned logic rather than framework-owned baseline data
